# Rail 5 File Delivery — Plan & Specification

**March 6, 2026 • Internal • v1**

---

## Problem Statement

When an owner connects a Rail 5 (encrypted) card to their bot, CreditClaw needs to deliver an encrypted card file to the bot. Currently, this delivery only works via webhook — but most bots don't have a webhook endpoint set up, and there's no way to add one after registration.

The bot that was tested confirmed this directly: it registered without a `callback_url`, has no web server running, and would need to re-register from scratch to receive webhooks. This is friction we need to eliminate.

---

## Goals

1. Make Rail 5 file delivery work for **every bot**, regardless of whether it has a webhook.
2. Give owners **multiple delivery options** so they can choose what works for their situation.
3. Bundle everything the bot needs into a **single, self-contained file** — encrypted card data, decrypt script, and usage instructions.
4. Track **webhook readiness** on each bot so the dashboard can show the right options.
5. Keep the existing webhook delivery path intact for bots that have one.

---

## The Single File Format

Instead of delivering three separate files (encrypted card, decrypt.js, instructions), everything is bundled into **one self-contained markdown file**.

### Why a Single File

- **One file to manage.** Owner downloads one file, bot saves one file.
- **Self-documenting.** If the bot's memory gets wiped or a new bot picks this file up later, the instructions are right there at the top.
- **Decrypt script is embedded.** No separate file to lose or misplace.
- **Links to the full skill.** Quick start covers the basics, then points to `skill.md` for the full Rail 5 checkout flow.

### File Structure

```markdown
# CreditClaw — Encrypted Card: Visa ending 4532
# Card ID: r5card_abc123

## How To Use This File

This file contains everything you need to perform Rail 5 encrypted card checkouts.
Save this file to your workspace (e.g., `.creditclaw/cards/`). Do not edit it.

### Quick Start

1. Save this entire file to `.creditclaw/cards/Card-Visa-4532.md`
2. Confirm delivery: `POST /api/v1/bot/rail5/confirm-delivery` with your API key
3. When ready to buy something, call `POST /api/v1/bot/rail5/checkout` with merchant details
4. You'll get a `checkout_id` — use it to get the decryption key from `POST /api/v1/bot/rail5/key`
5. Extract and run the decrypt script below with the key to get card details
6. Complete checkout at the merchant
7. Confirm result: `POST /api/v1/bot/rail5/confirm` with success or failure

### Running the Decrypt Script

Extract the JavaScript between the `DECRYPT_SCRIPT_START` and `DECRYPT_SCRIPT_END` markers below.
Save it as `decrypt.js`, then run:

    node decrypt.js <key_hex> <iv_hex> <tag_hex> Card-Visa-4532.md

The key, IV, and tag are provided by the `/api/v1/bot/rail5/key` endpoint during checkout.

### Full Documentation

For complete Rail 5 details, checkout flows, sub-agent mode, and API reference:
https://creditclaw.com/skill.md#rail-5

## Decrypt Script

<!-- DECRYPT_SCRIPT_START -->
```js
const crypto = require("crypto");
const fs = require("fs");
const [,, keyHex, ivHex, tagHex, filePath] = process.argv;

const raw = fs.readFileSync(filePath, "utf8");
const b64 = raw.match(/<!-- ENCRYPTED_CARD_START -->\n```\n([\s\S]+?)\n```\n<!-- ENCRYPTED_CARD_END -->/)[1].trim();
const data = Buffer.from(b64, "base64");

const decipher = crypto.createDecipheriv(
  "aes-256-gcm",
  Buffer.from(keyHex, "hex"),
  Buffer.from(ivHex, "hex")
);
decipher.setAuthTag(Buffer.from(tagHex, "hex"));
const plain = decipher.update(data.slice(0, -16)) + decipher.final("utf8");
process.stdout.write(plain);
```
<!-- DECRYPT_SCRIPT_END -->

## Encrypted Card Data

<!-- ENCRYPTED_CARD_START -->
```
<base64 encoded AES-256-GCM ciphertext>
```
<!-- ENCRYPTED_CARD_END -->
```

### Technical Notes on the Single File

- The decrypt script regex changes from the current `/```([\s\S]+?)```/` to target the specific `ENCRYPTED_CARD_START/END` markers. This avoids ambiguity since the file now contains multiple code fences.
- The `DECRYPT_SCRIPT` constant in `lib/rail5/decrypt-script.ts` needs to be updated to use the new marker-based regex.
- The `buildEncryptedCardFile()` function in `lib/rail5/encrypt.ts` needs to generate the new combined format instead of the current minimal format.
- Backward compatibility: existing encrypted card files (old format with a single code fence) should still work with the current decrypt script. The new format only applies to newly created cards.

## Four Delivery Options

When the owner connects a Rail 5 card on the dashboard, they see delivery options based on the bot's webhook status.

### Option 1: Push via Webhook

**When available:** Bot has a `callback_url` and `webhook_secret` configured.

**Flow:**
1. Owner clicks "Deliver to Bot" on the dashboard
2. CreditClaw sends the single file content via the existing `rail5.card.delivered` webhook event
3. Bot receives it, saves to `.creditclaw/cards/`, confirms delivery
4. Card status moves from `pending_delivery` → `confirmed`

**Changes needed:** Update the `deliver-to-bot` route to send the new combined file format instead of separate `encrypted_file_content` + `decrypt_script` fields. The webhook payload becomes:

```json
{
  "event": "rail5.card.delivered",
  "bot_id": "bot_abc123",
  "data": {
    "card_id": "r5card_abc123",
    "card_name": "Visa ending 4532",
    "card_last4": "4532",
    "file_content": "<the entire combined markdown file>",
    "suggested_path": ".creditclaw/cards/Card-Visa-4532.md",
    "instructions": "Save the file_content to your workspace at the suggested_path. Then confirm delivery at POST /api/v1/bot/rail5/confirm-delivery."
  }
}
```

### Option 2: Owner Sends Message to Bot to Pull

**When available:** Bot has no webhook (or owner prefers this method).

**Flow:**
1. Owner sets up Rail 5 card on dashboard
2. CreditClaw stages the combined file for pickup (stored temporarily, tied to bot_id)
3. Dashboard shows the owner a short, copy-paste message:

> **Send this to your bot:**
>
> "Fetch your encrypted card file by calling `GET https://creditclaw.com/api/v1/bot/rail5/pending-deliveries` with your CreditClaw API key. Save the file to `.creditclaw/cards/`."

4. Owner pastes this message to their bot (chat, email, however they communicate)
5. Bot calls the endpoint, receives the combined file, saves it, confirms delivery
6. CreditClaw purges the staged file after confirmation (or after 24-hour expiry)

**New endpoint needed:**

```
GET /api/v1/bot/rail5/pending-deliveries
Authorization: Bearer <bot_api_key>
```

Response:
```json
{
  "deliveries": [
    {
      "card_id": "r5card_abc123",
      "card_name": "Visa ending 4532",
      "card_last4": "4532",
      "file_content": "<the entire combined markdown file>",
      "suggested_filename": "Card-Visa-4532.md",
      "suggested_path": ".creditclaw/cards/Card-Visa-4532.md",
      "staged_at": "2026-03-06T10:00:00Z",
      "expires_at": "2026-03-07T10:00:00Z"
    }
  ],
  "instructions": "Save each file to the suggested_path in your workspace. Then confirm delivery at POST /api/v1/bot/rail5/confirm-delivery."
}
```

**Database change:** New `rail5_pending_deliveries` table (or a `pending_file_content` column on `rail5_cards`) to hold the staged file temporarily.

### Option 3: Owner Downloads and Sends Files Manually

**When available:** Always (universal fallback).

**Flow:**
1. Owner sets up Rail 5 card on dashboard
2. Owner clicks "Download Card File"
3. Browser downloads the single combined markdown file
4. Owner gives the file to their bot (drops it in workspace, uploads it, etc.)
5. Bot saves it, confirms delivery

**Changes needed:** Update the existing `downloadEncryptedFile()` function to generate the new combined format. The download button already exists on the dashboard — it just needs to output the new file format.

### Option 4: Custom Instructions for Agents

**When available:** Always. Best for AI agents (as opposed to simpler bots).

**Flow:**
1. Owner sets up Rail 5 card on dashboard
2. Owner clicks "Download with Agent Instructions"
3. Browser downloads the same combined markdown file (instructions are already embedded in it)
4. Dashboard also shows a short instruction block the owner can copy alongside the file:

> **Instructions for your agent:**
>
> "I've set up an encrypted payment card for you on CreditClaw (Rail 5). Here's the encrypted card file — save it to `.creditclaw/cards/` in your workspace. The file contains everything you need: usage instructions, the decrypt script, and the encrypted card data. Read the top of the file for a quick start guide. For full details, see https://creditclaw.com/skill.md#rail-5"

5. Owner pastes the instructions + file to their agent
6. Agent reads the instructions, saves the file, confirms delivery

**Changes needed:** Dashboard UI to show the copy-paste instructions alongside the download button. The file itself is the same as Option 3 — the self-documenting format means the agent can figure it out from the file alone, but the owner's message gives it extra context.

## Webhook Status Tracking

### Current State

The bot record already has `callbackUrl` (nullable) and `webhookSecret` (nullable). If both are present, the bot can receive webhooks. If either is null, it can't.

### Proposed Approach

No new database column needed. The webhook readiness check is simply:

```typescript
const hasWebhook = Boolean(bot.callbackUrl && bot.webhookSecret);
```

This is used in:
- **Dashboard:** To decide which delivery options to show (Option 1 only appears if `hasWebhook` is true)
- **Deliver-to-bot route:** Already checks this (returns 422 if missing)
- **Bot status endpoint:** Could optionally include `webhook_configured: true/false` in the response so the bot knows its own status

### Future: Deferred Webhook Setup

A separate future enhancement could add `POST /api/v1/bot/webhook/setup` to let bots register a webhook after initial registration. This is **not** part of this plan — the four delivery options above work without it. But it would be a natural follow-up that makes Option 1 available to more bots.

## Implementation Tasks

### Task 1: Update the Combined File Format

**Files:**
- `lib/rail5/encrypt.ts` — Update `buildEncryptedCardFile()` to generate the new combined format with instructions, embedded decrypt script, and marker-delimited encrypted data
- `lib/rail5/decrypt-script.ts` — Update `DECRYPT_SCRIPT` constant to use the new marker-based regex (`ENCRYPTED_CARD_START/END`)
- `lib/rail5/index.ts` — If any helper functions reference the old format

**Acceptance:** New cards produce a single self-contained markdown file with instructions at the top, decrypt script in the middle, and encrypted data at the bottom. Existing decrypt flow still works.

### Task 2: Build the Pending Deliveries Endpoint

**Files:**
- `shared/schema.ts` — Add `rail5PendingDeliveries` table (botId, cardId, fileContent, stagedAt, expiresAt, status)
- `server/storage.ts` — Add CRUD methods for pending deliveries
- `app/api/v1/bot/rail5/pending-deliveries/route.ts` — New GET endpoint, authenticated via `withBotApi`

**Acceptance:** Bot can call `GET /api/v1/bot/rail5/pending-deliveries` with its API key and receive any staged files. Files expire after 24 hours.

### Task 3: Update the Deliver-to-Bot Route

**Files:**
- `app/api/v1/rail5/deliver-to-bot/route.ts` — Update webhook payload to use new combined `file_content` field instead of separate `encrypted_file_content` + `decrypt_script`

**Acceptance:** Webhook delivery sends the new combined file format. Bot receives one `file_content` field with everything.

### Task 4: Stage Files for Pickup (No-Webhook Path)

**Files:**
- `app/api/v1/rail5/stage-for-pickup/route.ts` — New endpoint called by the dashboard when owner chooses "Stage for Pickup"
- Uses the same combined file format

**Acceptance:** When an owner stages a file, it appears in the bot's `pending-deliveries` response. Dashboard shows the copy-paste message.

### Task 5: Dashboard Delivery UI

**Files:**
- The Rail 5 card setup wizard component — Add the delivery options step showing all four options
- Webhook status check to determine which options to show
- Copy-to-clipboard for the pickup message
- Download button generates the new combined file format

**Acceptance:** Dashboard shows the appropriate delivery options based on webhook status. All four options produce the correct file/message.

### Task 6: Update Skill.md

**Files:**
- `public/skill.md` — Update the Rail 5 section to document the new file format and the pending-deliveries endpoint. Add guidance for bots without webhooks.

**Acceptance:** Skill.md reflects all four delivery paths. Bots without webhooks know they can poll `pending-deliveries`.

### Task 7: Cleanup and Expiry

**Files:**
- Cleanup logic (cron or middleware-triggered) to purge expired pending deliveries older than 24 hours
- `confirm-delivery` route should also purge the pending delivery record

**Acceptance:** Staged files are automatically cleaned up. Confirmed deliveries are removed from pending.

## Migration & Backward Compatibility

- **Existing encrypted card files** (old format with a single code fence and no embedded instructions) continue to work with the current decrypt script.
- **New cards** use the combined format with markers. The updated decrypt script handles the new marker-based format.
- **Existing webhook deliveries** continue to work — the payload just includes the new combined file format instead of separate fields.
- **No database migration for existing cards.** Only new pending delivery staging requires a new table.

---

## Security Considerations

- **Pending deliveries are authenticated.** Only the bot with the correct API key can retrieve its own staged files.
- **Files expire.** Staged content is purged after 24 hours, limiting the exposure window.
- **No plaintext card data.** The staged file contains only the encrypted ciphertext — CreditClaw still never holds both the ciphertext and the key.
- **The combined file doesn't weaken the split-knowledge model.** The file contains encrypted data and the decrypt *script*, but not the decrypt *key*. The key is still only available via the `/bot/rail5/key` endpoint during an active checkout.
- **Owner-initiated delivery.** All four options are triggered by the owner, maintaining human-in-the-loop control.
