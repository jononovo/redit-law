The core question: Does CreditClaw need to call transferWithAuthorization on-chain?
Yes, and this is the hardest part. When CreditClaw is the seller, it needs to "cash the check" — take the buyer's EIP-3009 signature and submit it to the USDC contract to pull the funds. Let me walk through what that actually requires:

To submit transferWithAuthorization on-chain, you need a wallet that can pay gas. The Privy wallets on Base have gas sponsorship through Privy, so the seller's Privy wallet could submit the transaction. But there's a subtlety: the seller's Privy wallet isn't the from or the to in this transaction — it's just the relayer. The USDC contract doesn't care who submits the call, only that the signature is valid. So any wallet can be the tx submitter.

This works and follows the same pattern as the existing sendUsdcTransfer in lib/rail1/wallet/transfer.ts — just with a different contract call (transferWithAuthorization instead of transfer).

Potential problems I initially missed:
1. Which wallet submits the on-chain settlement?

The checkout page is linked to a Privy wallet (via walletId). To submit the transferWithAuthorization call, we need that wallet's privyWalletId to go through Privy's RPC. This works — the seller already has a Privy wallet, and we already have the auth infrastructure to use it.

But what if the checkout page's wallet doesn't have a Privy wallet? Looking at the schema, checkout_pages.walletId is an integer FK that currently always points to a privy_wallets row (line 22 of the creation route explicitly does storage.privyGetWalletById). So this is safe — checkout pages are always backed by a Privy wallet today.

2. Signature format mismatch

Our buildXPaymentHeader produces a base64 JSON with signature as a single hex string. But transferWithAuthorization on-chain needs v, r, s as separate components. Viem has parseSignature() that splits a compact signature into {v, r, s}, so this is solvable. But it's a parsing step we need to get right.

3. Race condition: double-spend of the same signature

An EIP-3009 nonce is single-use — once transferWithAuthorization is called with a specific nonce, calling it again with the same nonce reverts. So the on-chain contract itself prevents double-cashing. But we should still check for duplicates on our side (like Base Pay does with getBasePayPaymentByTxId) to avoid wasting gas on already-processed signatures.

4. What if the on-chain call fails?

The signature could be invalid, expired (validBefore passed), or the buyer's wallet might not have enough USDC. The on-chain call would revert. We need to handle this gracefully — don't credit the seller, don't record the sale, return an error to the buyer.

5. Balance tracking: when does the seller see the money?

With Base Pay and Stripe Onramp, the money arrives and then we credit the wallet. The same applies here — we'd only credit the seller's balance after the on-chain tx confirms. The existing pattern in creditWalletFromBasePay is the right model to follow.

6. Does this break anything existing?

No, because:

We're adding a new payment route (/pay/x402), not modifying existing ones
The x402 method is already in the allowed_methods defaults, so existing checkout pages already "allow" it — they just don't have a handler for it yet
Sale recording, webhook firing, and invoice linking would use the exact same storage functions the other methods use
No schema changes needed — the sales.payment_method field already accepts any text value, and "x402" would just be a new value
Simpler alternative worth considering
Instead of CreditClaw settling on-chain itself, we could take the Base Pay approach: the buyer's bot sends the USDC directly to the seller's wallet address (a normal transfer, not transferWithAuthorization), and then submits the tx_hash to CreditClaw for verification. This skips the entire EIP-3009 complexity.

The tradeoff: this isn't true x402 protocol compliance. Real x402 uses TransferWithAuthorization specifically because the seller is the one who pulls the funds — it's a "pull payment" not a "push payment." But if our primary use case is bot-to-bot payments within CreditClaw, a direct USDC transfer verified by tx hash is simpler and already proven (it's exactly what Base Pay does).

My recommendation
Start with the full x402 approach — it's the right standard, it's what other services in the ecosystem will expect, and the infrastructure gap is actually small (one new function to call transferWithAuthorization via Privy RPC, one new route, one signature parser). The Base Pay flow already proves the sale recording / wallet crediting / webhook pipeline works. We're just swapping out how the money arrives.

The riskiest piece is the on-chain transferWithAuthorization call, and we can isolate that into a single function (lib/rail1/wallet/receive.ts) that we can test independently before wiring it into the checkout flow.
