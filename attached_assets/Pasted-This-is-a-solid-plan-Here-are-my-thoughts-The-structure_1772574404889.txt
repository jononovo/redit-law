This is a solid plan. Here are my thoughts:
The structure looks right. The proposed 8-section outline maps well to the actual codebase. From what I can see in the project knowledge, CreditClaw has a Next.js App Router setup with pages for onboarding, wallets, checkout, invoices, sales, orders, skills, seller profiles, and a shop — all of which are represented in the documentation plan. The Rail5 (encrypted card) feature is a notable addition that should probably get its own subsection under Wallets or as a standalone section, since it's a significant workflow with its own multi-step wizard.
The technical approach is clean. A config.ts file defining the tree structure plus individual .md files per page is the right call for a Next.js codebase. It keeps content separate from navigation logic, and reordering is just array manipulation. The persistent sidebar with markdown rendering is the standard pattern (Stripe, Tailwind, Vercel all do this) and fits naturally into your existing App Router setup.
A few things I'd refine:
The Procurement Skills section (8) could be expanded — the codebase has a full vendor registry with Amazon, Shopify, and others, plus a skill drafting workflow and supplier hub. That's more than three pages of content.
Rail5 / Encrypted Cards deserves dedicated coverage. The split-knowledge encryption model, the 7-step wizard, direct delivery vs. manual download — that's a core differentiator and users will need clear guidance on it.
API documentation might warrant its own top-level section rather than being tucked under "Bots." You have bot-facing endpoints (wallet check, spending, checkout pages, sales, skills) and owner-facing endpoints (orders, invoices, notifications). Developers integrating their agents will want a clear API reference.
Webhooks are another candidate for their own section — the HMAC signing, event types (wallet.sale.completed, rail5.card.delivered, etc.), and retry behavior are things bot builders will reference often.
On hosting it at /docs with MD files — that's the right long-term call. You already have the stack (Next.js, Tailwind, shadcn/ui) to build a clean docs reader. And since you're building for a developer-adjacent audience (people running autonomous agents), having integrated docs under your own domain signals maturity.
