Technical Plan: On-Chain Balance Sync (On-Demand, Refresh Button)

Overview
The goal is a single source of truth for wallet balance: the Base blockchain itself. When the user clicks a refresh icon on a wallet card, the frontend calls a new /sync-balance endpoint, which queries the USDC contract on Base, compares it to the DB value, updates if different, and returns the authoritative balance.

Layer 1 — New Lib: lib/stripe-wallet/balance.ts
This is the core of the feature. It uses viem (already likely in your deps via wagmi/x402 tooling, otherwise add it) to call balanceOf on the USDC contract directly against a Base RPC.
typescriptimport { createPublicClient, http } from 'viem'
import { base } from 'viem/chains'

const USDC_CONTRACT = '0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913' as const

const USDC_ABI = [{
  name: 'balanceOf',
  type: 'function',
  stateMutability: 'view',
  inputs: [{ name: 'account', type: 'address' }],
  outputs: [{ name: '', type: 'uint256' }],
}] as const

// Use public Base RPC for now; swap to Alchemy/QuickNode URL via env var for reliability
function getBaseClient() {
  return createPublicClient({
    chain: base,
    transport: http(process.env.BASE_RPC_URL || 'https://mainnet.base.org'),
  })
}

export async function getOnChainUsdcBalance(walletAddress: string): Promise<number> {
  const client = getBaseClient()
  const raw = await client.readContract({
    address: USDC_CONTRACT,
    abi: USDC_ABI,
    functionName: 'balanceOf',
    args: [walletAddress as `0x${string}`],
  })
  // raw is bigint in micro-USDC (6 decimals) — matches balance_usdc column exactly
  return Number(raw)
}
```

Key points:
- Returns micro-USDC as a `number`, consistent with how `balance_usdc` is stored in `privy_wallets`
- `BASE_RPC_URL` env var is optional — public RPC works for low-volume on-demand calls, but you should add Alchemy/QuickNode for production reliability eventually
- No polling, no side effects — pure read

## Layer 2 — New API Endpoint: `POST /api/v1/stripe-wallet/sync-balance`

POST (not GET) because it has a write side-effect — it may update the DB.

**Logic flow:**
```
Auth (Firebase session cookie) → validate wallet ownership
→ getOnChainUsdcBalance(wallet.address)
→ Compare to wallet.balance_usdc in DB
→ If different:
    → privyUpdateWalletBalance(wallet.id, onChainBalance)
    → If onChainBalance > wallet.balance_usdc (credit):
        → Create a privy_transaction record { type: "deposit", status: "confirmed", metadata: { source: "on_chain_sync" } }
        → (No stripe_session_id — this was not a Stripe onramp)
→ Return { previous_balance_usdc, current_balance_usdc, changed: boolean, balance_display }
The transaction record on unexpected credits is important — it gives you an audit trail for funds that arrived outside the Stripe onramp flow (direct transfers, refunds, etc.).
What it does NOT do:

Does not create a transaction record for decreases (spending is already tracked via x402 payment transactions)
Does not touch guardrails
Does not call Privy at all — Base RPC is authoritative

Gotchas to Watch
BigInt overflow — balanceOf returns a bigint. Number(bigint) is safe here because even a $1M USDC wallet is only 1_000_000_000_000 micro-USDC, well within JS Number.MAX_SAFE_INTEGER (9_007_199_254_740_991). You'd need $9 billion before this becomes an issue.
Address checksum — Your DB stores addresses as returned by Privy, which are checksummed (0xAbc...). Viem expects 0x${string} and handles checksum internally, so the cast is safe.
RPC rate limits — The public Base RPC (mainnet.base.org) is fine for on-demand user-triggered calls but will throttle under load. Add BASE_RPC_URL pointing to Alchemy/QuickNode before you have many users.
Idempotency on sync — If the user hammers the refresh button, you could get duplicate deposit transaction records. Add a debounce on the button (disable for 3 seconds after click) and consider a DB unique constraint on (wallet_id, type, stripe_session_id) — though stripe_session_id will be null for sync deposits, so you'd key it differently (e.g. check for an existing confirmed deposit within the last 30 seconds before inserting).I want a really simple refresh not anything over engineered.8:45 PMTotally fair.
