Phase 1 — Unified Bot Status Endpoint (Must-Have, Solves the Immediate Problem)
Endpoint: GET /api/v1/bot/status

Simplified response:

{
  "bot_id": "bot_876b1560",
  "bot_name": "bennie",
  "wallet_status": "active",
  "default_rail": "rail4",
  "rails": {
    "rail1": { "status": "active", "balance_usd": 45.00 },
    "rail2": { "status": "inactive" },
    "rail4": { "status": "active", "card_name": "Business Visa", "last4": "4532" },
    "rail5": { "status": "inactive" }
  },
  "master_guardrails": {
    "max_per_tx_usd": 100,
    "daily_budget_usd": 1000,
    "monthly_budget_usd": 10000,
    "daily_spent_usd": 23.50,
    "monthly_spent_usd": 147.00
  }
}
Tasks:

Add default_rail column to bots table (nullable text, defaults to null — system auto-picks first active rail if unset)
Create app/api/v1/bot/status/route.ts using withBotApi middleware
Query all 4 rail tables in parallel using existing storage methods
Query master guardrails config + actual daily/monthly spend totals
Return flat, clean response — balance_usd everywhere (no USDC distinction), objects per rail (not arrays), inactive rails return { "status": "inactive" }
Update skill docs (SKILL.md, heartbeat.md) so bots know to call /bot/status instead of /bot/wallet/check
Add default rail dropdown to dashboard settings page
No schema changes needed beyond the default_rail column. All storage methods already exist.

Tasks:

Create shared rail-status builder utility (reuses Phase 1 logic, avoids duplication)
Create 4 route files under app/api/v1/bot/check/
Each returns rail-specific operational detail — allowance remaining, reset dates, approval mode
