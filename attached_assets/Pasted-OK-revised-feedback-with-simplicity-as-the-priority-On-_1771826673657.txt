OK, revised feedback with simplicity as the priority.

On the Response Shape
Collapsing /status to just return master guardrails plus the rails overview is the right call. The bot doesn't need two competing limit objects. Master guardrails are the ceiling, per-rail limits are handled when the bot actually tries to spend. Keep it simple.
On USD vs USDC — just use balance_usd everywhere. USDC is pegged 1:1, the bot doesn't care about the distinction, and having two currency fields creates confusion for no practical benefit. If you ever add a non-dollar-pegged currency you can revisit, but for now one field is correct.
So the revised /bot/status response would look something like:
json{
  "bot_id": "bot_876b1560",
  "bot_name": "bennie",
  "wallet_status": "active",
  "default_rail": "rail4",
  "rails": {
    "rail1": { "status": "active", "balance_usd": 45.00 },
    "rail2": { "status": "inactive" },
    "rail4": { "status": "active", "card_name": "Business Visa", "last4": "4532" },
    "rail5": { "status": "inactive" }
  },
  "master_guardrails": {
    "max_per_tx_usd": 100,
    "daily_budget_usd": 1000,
    "monthly_budget_usd": 10000,
    "daily_spent_usd": 23.50,
    "monthly_spent_usd": 147.00
  }
}
Clean, flat, one call. The bot knows what rails it has, what the ceiling is, and how much headroom it has. That's it.

On the Per-Rail Detail Endpoints
GET /bot/check/rail4 is where the bot gets operational detail for that rail — allowance remaining, allowance reset date, approval mode. Keep it thin. Don't try to return transaction history or obfuscation data here. Just the stuff the bot needs to decide "can I make this purchase and will it need human approval?"
For Rail 4 specifically, the obfuscation layer question is simpler than I made it sound. The bot already interacts with profiles through profile_index in the checkout request. The /check/rail4 response just needs to return whatever the bot already knows from its decoy file — which profile indices exist and what the allowance status is for each. The bot can't distinguish real from fake, and that's the point. Don't overthink this.

On What's Still Missing but Important
Skill doc updates. This is the thing that will actually make or break whether the bot uses the new endpoints. The bot reads skill.md and heartbeat.md to learn what API calls to make. If those still say "call /bot/wallet/check" then nothing changes from the bot's perspective. When you build Phase 1, updating the skill files should be part of the same work, not a follow-up.
Rate limits. Just match what you have — /bot/status at 6/hr (same as the current /bot/wallet/check it replaces), per-rail checks at 6/hr. Nothing fancy.
default_rail in the database. You'll need a column somewhere — probably on the bots table or a new bot settings table. Dashboard UI to set it. Small but needs to exist before /bot/status can return it. Could default to the first active rail if not explicitly set.

On the Plan Overall
The phasing is right. Phase 1 (/bot/status) solves the actual problem your bot is hitting today. Phase 2 (per-rail checks) gives depth when needed. Phase 3 (preflight/test) is genuinely useful but not urgent.
The one-active-per-rail rule keeps everything simple and I'd stick with it. Just make sure the response structure uses objects (not arrays) for each rail so you don't have to restructure if you ever relax that constraint — an object with one card's data can become an array of cards later without breaking consumers.
The plan is good. Build Phase 1, update the skill docs as part of it, and you'll have solved the core problem.
